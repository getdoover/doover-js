# doover-js 0.5.0 tidyup — design

**Status:** approved (brainstorming)
**Target version:** 0.5.0
**Removal of deprecated surface:** 0.6.0
**Date:** 2026-05-07

## Background

`DooverDataProvider` (the "viewer") is currently the centre of the package — it owns the gateway, holds subscription state, drives RPC, normalises agent responses, and re-wraps most of the `*Api` classes. `DooverClient` was added later as a broader facade but the React hooks and most internal logic still go through `client.viewer.X`.

The result is two coexisting public surfaces over the same backend, two `RestClient` instances per `DooverClient`, two sets of resource API instances, and a stateful coordinator (the viewer) that mixes pure REST passthroughs with subscription bookkeeping and an RPC-over-messages flow.

0.5.0 reorganises the package into three honest layers — transport, resources, stateful coordinator — and demotes the viewer to a thin deprecated wrapper.

## Goals

1. `*Api` classes are the canonical surface for REST resource access.
2. Subscription bookkeeping lives on `GatewayClient`.
3. RPC-over-messages becomes a dedicated `RpcDispatcher` class with timeout, abort, typed errors, and instrumentation.
4. One `RestClient` per `DooverClient`.
5. React hooks consume the new surface; no hook references `client.viewer`.
6. `DooverDataProvider` becomes a thin deprecated delegating wrapper, removed in 0.6.0.

## Non-goals

- Generating `*Api` classes from the openapi spec.
- Reworking WS reconnect or auth flows.
- Adding peak-pending counters to `rest` / `gateway` stats (only `rpc` gets one).
- Refactoring `AuthProfile` / `ConfigManager`.
- Splitting React hooks into a separate package.

## Architecture

```
DooverAuth                    one instance per DooverClient
   │
   ├── RestClient             one instance (was two)
   │      └── *Api classes    canonical surface; outer/inner pattern
   │
   ├── GatewayClient          gains channel-subscription registry
   │
   └── RpcDispatcher          new; depends on (gateway, messagesApi)

DooverClient                  facade; wires above; exposes
                                .auth .rest .gateway .rpc .stats
                                .users
                                .agents .channels .messages .aggregates
                                .alarms .connections .notifications
                                .permissions .processors .turn
                                .viewer  (deprecated thin wrapper)
```

## Components

### `*Api` classes — canonical resource surface

Each method that takes `agentId` (with optional `channelName`) uses the **outer/inner pattern**:

- **Outer** (public): two overload signatures — strict positional, or an identifier object (`ChannelIdentifier` / `{ agentId }`). Body delegates to the inner method.
- **Inner** (`_method`, private): strict positional. Holds the actual `rest.*` call. The inner is the canonical implementation; the outer is a pure adapter.
- A shared `resolveChannelArgs` / `resolveAgentArgs` helper in `src/apis/_args.ts` does the narrowing once. Each outer becomes one line of narrowing plus the delegation.

Worked example:

```ts
type ChannelIdentifier = { agentId: string; channelName: string };

export class ChannelsApi {
  constructor(private readonly rest: RestClient) {}

  getChannel(agentId: string, channelName: string, options?: GetChannelOptions): Promise<Channel>;
  getChannel(identifier: ChannelIdentifier, options?: GetChannelOptions): Promise<Channel>;
  getChannel(...args: unknown[]): Promise<Channel> {
    const { agentId, channelName, options } = resolveChannelArgs<GetChannelOptions>(args);
    return this._getChannel(agentId, channelName, options);
  }

  private _getChannel(
    agentId: string,
    channelName: string,
    options?: GetChannelOptions,
  ): Promise<Channel> {
    return this.rest.get<Channel>(`/agents/${agentId}/channels/${channelName}`, options);
  }
}
```

**Logic absorbed from the viewer:**

- `AgentsApi.listAgents({ includeArchived?, includeOrganisations?, includeUsers?, mergeIncludedAsAgents? })` — owns `normalizeAgentEntry`, `normalizeOrganisationEntry`, `normalizeUserEntry`, and the merge logic that produces the combined `agents` / `results` array.
- `MessagesApi.listMessages` gains:
  - default `limit: 10`
  - default `before` = snowflake-id of "now" when omitted
  - new option `order?: "asc" | "desc"` (default `"desc"`, matching server order). The deprecated viewer wrapper passes `order: "asc"` to preserve today's `.reverse()` behaviour.
- `ChannelsApi.archiveChannel(id)` and `unarchiveChannel(id)` — currently the viewer hits `/archive` and `/unarchive` directly via `rest.post`. Move into `ChannelsApi`.
- `MessagesApi.deleteMessage` and `postMessage` — already exist; the viewer wrappers go away.

**New small surface:**

- `UsersApi` with one method: `getMe()` → `/users/me`. Currently the viewer calls this directly via `rest.get`. One method is enough to warrant a tiny class for symmetry with the other resource APIs.

**Stays out of the new surface:**

- `getAgentInfo` is a synthesized stub today (no network call). Dropped with no replacement.
- `getAggregate` cross-resource fallback (`channelsApi.getChannel` → `aggregatesApi.getAggregate` if absent) stays only on the deprecated viewer. The convenience disappears with 0.6.0.

### `GatewayClient` additions

```ts
type ChannelKey = { agent_id: string; name: string };
type ChannelHandlers = {
  onMessage?: (msg: MessageStructure) => void;
  onMessageUpdate?: (msg: MessageStructure, requestData?: JSONValue) => void;
  onAggregate?: (agg: Aggregate) => void;
};

class GatewayClient {
  // existing: connect(), subscribe(), unsubscribe(), on(), sendOneShotMessage(), ...

  subscribeToChannel(channel: ChannelKey, handlers: ChannelHandlers): () => void;
}
```

**Behaviour:**

- One internal `Map<channelKey, Set<ChannelHandlers>>`.
- First handler for a given channel triggers the wire-level `subscribe(...)`.
- Last handler removed triggers wire-level `unsubscribe(...)`.
- Returned function is the canonical detach handle. Idempotent — calling it twice is a no-op.
- The four `gateway.on(...)` listeners currently in the viewer constructor (`messageCreate`, `messageUpdate`, `channelSync`, `aggregateUpdate`) move into the gateway and fan out to the registry.

The callback-ref-as-identity pattern used by the current viewer (`unsubscribeFromChannel(id, callback)` matched by reference) is gone from the new surface. The deprecated viewer wrapper preserves the old signature by holding a small `Map<callback, () => void>` that translates to the new shape.

### `RpcDispatcher` — new class

```ts
class RpcDispatcher {
  constructor(gateway: GatewayClient, messages: MessagesApi);

  send<TReq, TRes, TPending = undefined>(
    channel: ChannelIdentifier,
    request: RpcRequest<TReq>,
    options?: {
      onStatus?: (status: RpcStatus<TPending>) => void;
      signal?: AbortSignal;
      timeoutMs?: number;
    },
  ): Promise<TRes>;

  setStats(collector: DooverStatsCollector): void;
}
```

**State:**

```ts
pending:      Map<messageId, { resolve, reject, onStatus, timer?, channelKey, startedAt }>
channelRefs:  Map<channelKey, { unsubscribe: () => void; refCount: number }>
```

**`send` flow:**

1. Look up `channelKey` in `channelRefs`.
   - If absent: `gateway.subscribeToChannel(channel, { onMessageUpdate: this.route })`, store the returned unsubscribe with `refCount: 1`.
   - If present: increment `refCount`.
2. Record stats start; obtain `startedAt` timestamp.
3. `messages.postMessage(agentId, channelName, { type: "rpc", ...request })` (REST).
4. Store `pending.set(messageId, { ..., channelKey, startedAt })`.
5. If `signal` provided: register `signal.addEventListener("abort", ...)` to settle as `abort`.
6. If `timeoutMs` provided: schedule timer to settle as `timeout`.

**`route` (one persistent handler, registered in constructor):**

- `pending.get(msg.id)` — if absent, ignore.
- If `msg.data` is `RpcMessageData` → `onStatus(status)`.
- `status.code === "success"` → resolve with `response`, settle.
- `status.code === "error"` → reject with `new DooverRpcError(status.message, status)`, settle.
- Other statuses (`sent`, `acknowledged`, `deferred`, `pending`) only fire `onStatus`.

**Settlement (success / error / timeout / abort):**

1. `pending.delete(messageId)`, clear timer, remove signal listener.
2. Decrement the channel's `refCount`. If it reaches `0`: call `unsubscribe()` and delete the channel entry.
3. Record stats end with the corresponding outcome.

**Concurrency:**

- N concurrent RPCs across any channels share one persistent gateway listener (`route`).
- N concurrent RPCs on the same channel share one dispatcher-side gateway subscription, refcounted.

**New error class:** `DooverRpcError extends Error` — wraps `status.message` plus original `status` and `request` for inspection. Replaces today's bare-string reject.

### `DooverClient` — facade

- Constructs one `RestClient` and passes it into the deprecated viewer.
- `client.gateway` is the only `GatewayClient` (still aliased through the viewer for back-compat — `client.viewer.gateway === client.gateway`).
- New: `client.rpc = new RpcDispatcher(this.gateway, this.messages)`.
- New: `client.users = new UsersApi(this.rest)`.
- `stats` is wired to one `RestClient`, the gateway, and the dispatcher.

### `DooverDataProvider` — deprecated thin wrapper

- Constructor receives `RestClient` and `GatewayClient` from `DooverClient`. The pre-0.5.0 constructor signature (which built its own transports from a `DooverClientConfig`) is retained as a deprecated overload to avoid breaking direct constructors.
- Holds **no** subscription state and **no** RPC state. All methods delegate to the canonical surface.
- `subscribeToChannel`/`unsubscribeFromChannel` keep their existing callback-ref signature, internally translated via a per-instance `Map<callback, () => void>` to bridge to the new gateway shape.
- Every method gets `@deprecated` JSDoc pointing at its replacement (see §"Deprecation map" below).
- README gains a "Migrating to 0.6.0" section with the same table.

## Data flow

### REST request

`client.channels.getChannel(...)` → outer narrows args → `_getChannel` → `RestClient.get(...)` → auth header → fetch → response. Stats counter increments on the single shared `RestClient`. Unchanged from today except for the single-RestClient consolidation.

### Channel subscription

```
caller
  └── client.gateway.subscribeToChannel({agent_id, name}, handlers)
         ├── if first handler for channel → gateway.subscribe() over WS
         └── returns () => void  (detach handle)

WS frame arrives
  └── GatewayClient parses → emits "messageCreate" / "messageUpdate" / "channelSync" / "aggregateUpdate"
         └── internal registry lookup by channel key
                └── fan out to each handler's onMessage / onMessageUpdate / onAggregate
```

### RPC over messages

```
client.rpc.send({agentId, channelName}, request, { onStatus?, signal?, timeoutMs? })
  ├── ensure channelRefs entry; subscribe if first
  ├── stats.recordRpcStart() → startedAt
  ├── messagesApi.postMessage(...)             ← REST
  │     └── returns { id: messageId }
  ├── pending.set(messageId, { ..., channelKey, startedAt })
  ├── attach abort/timeout listeners
  └── (already listening) gateway.on("messageUpdate", route)
        └── route(msg):
              ├── pending.get(msg.id) — if absent, ignore
              ├── if data is RpcMessageData → onStatus(status)
              ├── status.code === "success" → resolve(response); settle("success")
              └── status.code === "error"   → reject(new DooverRpcError(...)); settle("error")

signal.aborted   → settle("abort")   → reject(AbortError)
timeoutMs hit    → settle("timeout") → reject(timeout error)

settle(outcome):
  pending.delete; clear timer; remove signal listener
  channelRefs[channelKey].refCount--
    if 0 → unsubscribe() and delete channel entry
  stats.recordRpcEnd(startedAt, outcome)
```

The dispatcher requires a channel subscription because the gateway only delivers `messageUpdate` events for subscribed channels. Subscription is taken via `gateway.subscribeToChannel`, which gives wire-level refcounting for free.

### Stats flow

- One `RestClient` → one rest counter source.
- One `GatewayClient` → one gateway counter source.
- `RpcDispatcher` → new `rpc` counters (see "Stats" below). REST POSTs and WS frames continue to increment under `rest` and `gateway`, so the snapshot decomposes cleanly.

## Stats

New `RpcStatsSnapshot` field on `DooverStatsSnapshot`:

```ts
interface RpcStatsSnapshot {
  enabled: boolean;
  totalRpcs: number;
  pendingRpcs: number;
  completedRpcs: number;     // terminal "success"
  failedRpcs: number;        // terminal "error"
  timedOutRpcs: number;      // settled by timeoutMs
  abortedRpcs: number;       // settled by AbortSignal
  averageLatencyMs: number | null;  // send() → terminal status
  lastLatencyMs: number | null;
  peakPendingRpcs: number;   // high-water since last reset
}

interface DooverStatsSnapshot {
  rest: RestStatsSnapshot;
  gateway: GatewayStatsSnapshot;
  rpc: RpcStatsSnapshot;     // new
}
```

Collector additions matching the existing `recordRestStart` / `recordRestEnd` pattern:

```ts
recordRpcStart(): number | null;
recordRpcEnd(
  startedAt: number | null,
  outcome: "success" | "error" | "timeout" | "abort",
): void;
```

`recordRpcStart` increments `totalRpcs` and `pendingRpcs`, updates `peakPendingRpcs`, returns the start timestamp (or `null` when disabled). `reset()` gains the corresponding RPC clauses.

`peakPendingRpcs` is the only peak counter introduced; existing `rest` and `gateway` counter sets are unchanged.

## React hooks migration

All `client.viewer.X` references in `src/react/` are migrated to the new surface as part of 0.5.0. After the change, no React hook references `client.viewer`.

| File | Current | New |
|---|---|---|
| `useChannelMessages.ts:111` | `client.viewer.getMessages(id, opts)` | `client.messages.listMessages(id, { ...opts, order: "asc" })` |
| `useChannelAggregate.ts:90` | `client.viewer.getAggregate(id)` | `client.channels.getChannel(id)` reading `.aggregate`, with inline fallback to `client.aggregates.getAggregate(id)` |
| `useSendRpc.ts:225` | `client.viewer.sendRPC<...>(...)` | `client.rpc.send<...>(...)` |
| `useSendMessage.ts:18` | `client.viewer.sendMessage(id, data)` | `client.messages.postMessage(id, { data })` |
| `useUpdateAggregate.ts:38–39` | `client.viewer.putAggregate / updateAggregate` | `client.aggregates.putAggregate / patchAggregate` |
| `useChannelSubscription.ts:54` | `client.viewer.subscribeToChannel(...)` | `client.gateway.subscribeToChannel(channel, handlers)` — store returned unsubscribe in effect cleanup |
| `useMultiAgentAggregates.ts:121` | same | same |
| `useMultiAgentChannelMessages.ts:93` | same | same |

Subscription hooks currently track teardown by holding the original callback reference and calling `unsubscribeFromChannel(id, cb)`. They move to capturing the returned unsubscribe fn directly in the effect's return, eliminating the callback-ref-as-key pattern.

## Rollout

Single PR for 0.5.0. The change is internally cohesive — RPC depends on the new subscription registry, the React hook migration depends on the `*Api` overloads, and deprecating the viewer requires the React hooks to be migrated. Splitting would create intermediate states uglier than the current state.

Version bump: `0.4.5 → 0.5.0`. Public API gains; deprecated members still work at runtime. README adds the migration table.

## 0.6.0 removal plan

- Delete `src/viewer/doover-data-provider.ts`.
- Delete its export from `src/index.ts`.
- Delete the `viewer` field on `DooverClient`.
- Delete the deprecated viewer test file.
- README migration section moves to the changelog.

The removal is mechanical; nothing on the deprecated wrapper has logic that isn't already in the new surface.

## Migration window expectations

- **SDK consumers using `client.viewer.X`:** 0.5.0 is non-breaking at runtime. TypeScript surfaces `@deprecated` strikethrough; `tsc --strict` does not error.
- **SDK consumers constructing `DooverDataProvider` directly:** still works; the pre-0.5.0 constructor signature is retained as a deprecated overload.
- **SDK consumers extending `DooverDataProvider`:** not officially supported. Flag in the migration note.

## Out of scope

- Generating `*Api` classes from the openapi spec.
- Reworking WS reconnect/auth flows.
- Adding peak-pending counters to `rest` / `gateway` stats.
- Refactoring `AuthProfile` / `ConfigManager`.
- Splitting React hooks into a separate package.

Adjacent cleanups discovered during implementation get filed as follow-up issues rather than absorbed.
